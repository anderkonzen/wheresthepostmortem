## Problem Statement

A developer in Slack asked: "Is there any sort of website that collects postmortems / incident summaries? if not, might be a fun project.... I learn lots about things to watch out for by reading some of these."

Existing postmortem collections (danluu/post-mortems on GitHub, SRE Weekly, company blogs) are either raw link dumps with zero curation or company-specific. The tools ecosystem (incident.io, Rootly, PagerDuty) helps you write postmortems, not read others'. Nobody has built a curated, beautifully presented reading experience for incident stories aimed at the broader dev community.

## What Makes This Cool

The gap is obvious once you see it: 400+ links in a markdown file vs. zero craft. This project fills that gap with the opposite approach. Fewer entries, way more thought per entry. Long-term goal: 30-50 curated postmortems. v1 launch: 10-15 seed entries. The differentiation is taste and presentation, not comprehensiveness.

The reading experience IS the product. Postmortems are inherently dramatic stories, some of the most compelling writing in tech. A database engineer accidentally deletes production at midnight. A deploy bug loses $440M in 45 minutes. Five backup strategies all fail simultaneously. These deserve to be presented like the narratives they are, not bullet points in a README.

Secondary motivation: a vehicle for testing gstack's full skill pipeline (design, build, QA, ship).

## Constraints

- Static content site, no dynamic features, no user accounts, no community submissions
- Source material is publicly available postmortems (company blogs, GitHub repos, public docs)
- Curating what's public, not creating original analysis
- Must work well on mobile (developers read on phones)
- No external runtime dependencies. Fonts (Source Serif 4 + Inter) self-hosted and bundled with the static site (download from Google Fonts, serve from /fonts/). No CDN dependency at runtime.
- Accessibility: WCAG AA compliance minimum. All color pairings must meet 4.5:1 contrast for body text, 3:1 for large text. Keyboard navigable. Semantic HTML.
- Performance: Lighthouse Performance score >= 95. Total page weight < 200KB (excluding fonts). Fonts < 150KB (latin subset). LCP < 1.5s (measured on simulated 4G, which is Lighthouse default).

## Premises

1. Developers want to READ postmortems for learning, not just reference them during incidents.
2. Curation quality matters more than comprehensiveness. v1 target: 10-15 entries. Long-term: 30-50.
3. Presentation and reading experience is the differentiator over existing link-dump collections.
4. This is a static content site, not a dynamic platform with user accounts, submissions, or community features.
5. The content source is publicly available postmortems. Curating what's already public, not creating original content.

## Cross-Model Perspective

Independent Claude subagent review (cold read, no conversation context):

**Coolest version not yet considered:** An interactive "incident timeline" mode where readers replay incidents step-by-step with "what would you do next?" decision points. Think choose-your-own-adventure meets incident response. (Noted as a compelling v2 feature, not v1 scope.)

**Key insight:** "This isn't about postmortems. This is about craft. The postmortem domain is the excuse; the real excitement is building a content experience with taste in a space full of markdown dumps."

**50% shortcut:** Astro (astro.build) with content collections. Typed frontmatter schemas are literally built for this use case. The 50% you'd need to build: taxonomy design, editorial curation work, and bespoke UI components that make it feel like a magazine.

**Suggested starting point:** Astro + Tailwind + Vercel. Scaffold, seed 5 entries, build homepage + detail + about pages, then polish and deploy.

## Approaches Considered

### Approach A: Astro + MDX Content Site (Minimal Viable)
Astro static site with content collections. Postmortems as MDX files with typed frontmatter. Tailwind for styling. Deploy to Vercel. Minimal design investment.
- Effort: S | Risk: Low
- Pros: Zero JS by default, fast, SEO-friendly, dead simple to add entries
- Cons: Limited interactivity, dev workflow for content (no CMS)

### Approach B: Next.js + MDX with Richer Interactivity (Ideal Architecture)
Next.js App Router with MDX content. Client-side search (Pagefind/Fuse.js), tag filtering, reading progress.
- Effort: M | Risk: Medium
- Pros: Richer interaction, better search, grows into more features
- Cons: More JS, more setup, overkill for mostly static content

### Approach C: Astro + Magazine-Grade Design (Selected)
Astro like Approach A, but invest saved engineering time into design. Custom typography, editorial layout, distinctive visual identity. Code is simple, craft is in the pixels.
- Effort: M | Risk: Low-Medium
- Pros: Maximum differentiation on the actual thesis (presentation). Simple architecture.
- Cons: Design iteration is subjective, visual polish is the bottleneck.

## Recommended Approach

**Approach C: Astro + Magazine-Grade Design.**

The thesis is presentation. The code is the easy part. Invest where the differentiation lives.

### Tech Stack
- **Framework:** Astro with content collections
- **Content:** MDX files with typed frontmatter
- **Styling:** Tailwind CSS with custom design tokens (see below)
- **Fonts:** Source Serif 4 (editorial body/headings) + Inter (UI chrome), self-hosted from /fonts/. Use variable font versions (one WOFF2 per family instead of per-weight files). Latin subset only. `font-display: swap`. Target combined font payload < 100KB with variable fonts + subsetting.
- **Deploy:** Vercel (static output, auto-deploy on push to main)

### Design Direction
- **Theme:** Light, warm off-white (#faf8f5) background
- **Accent:** Warm amber (#b45309) for brand color, links, section headers
- **Typography:** Source Serif 4 at 600/700 weight for headings, 400 for body. Inter for meta, tags, filters.
- **Cards:** White (#fff) on off-white, subtle borders (#e8e3da), minimal hover effects
- **Detail page:** 700px reading column, 1.1rem body text, 1.8 line-height
- **Vibe:** Editorial warmth. Like reading a well-designed technical book. Not a dashboard, not a monitoring tool.

### Motion & Transitions

Minimal, intentional motion. This is an editorial site, not an app.

1. **Card hover:** `transition: border-color 150ms ease, box-shadow 150ms ease`. No transform/scale.
2. **Filter transition:** Cards fade in/out with `opacity` transition (150ms) when filtering. No layout shift.
3. **Page transitions:** None. Static site, instant navigation. Don't add artificial transitions.
4. **Scroll:** No scroll-linked animation. No parallax. Content sites should feel like reading, not performing.
5. **`prefers-reduced-motion`:** Respect the OS setting. Wrap all transitions in `@media (prefers-reduced-motion: no-preference)`.

**Accessibility check:** amber accent (#b45309) on off-white (#faf8f5) = 4.84:1 contrast ratio (passes WCAG AA). body text (#3a3a3a) on off-white = 10.5:1 (passes AAA). All severity badge colors verified AA compliant within their background tints.

### Content Schema Validation

Define the schema in `src/content/config.ts` using Astro's built-in Zod validation. Build fails on invalid frontmatter. Zero runtime cost.

```ts
// src/content/config.ts
import { defineCollection, z } from 'astro:content';

const postmortems = defineCollection({
  type: 'content',
  schema: z.object({
    title: z.string(),
    slug: z.string(),
    company: z.string(),
    date: z.date(),
    last_updated: z.date(),
    draft: z.boolean().default(false),
    featured: z.boolean().default(false),
    highlighted: z.boolean().default(false),
    severity: z.enum(['critical', 'major', 'minor']),
    categories: z.array(z.enum([
      'database', 'dns', 'deploy', 'cascading-failure', 'configuration',
      'security', 'networking', 'third-party', 'caching', 'load', 'financial'
    ])),
    root_cause_type: z.enum([
      'human-error', 'configuration-change', 'deploy-failure', 'cascading-failure',
      'security-breach', 'third-party-dependency', 'capacity', 'latent-bug'
    ]),
    one_line_summary: z.string(),
    source_url: z.string().url(),
    reading_time: z.number(),
  }),
});

export const collections = { postmortems };
```

### Content Schema (Frontmatter)
```yaml
title: "When a Database Engineer Accidentally Deleted the Production Database"
slug: "gitlab-database-deletion-2017"  # URL slug, must be unique
company: "GitLab"
date: 2017-01-31
last_updated: 2026-04-09  # when this summary was last verified
draft: false  # set true to hide from production build
featured: false  # set true for the one featured entry on homepage (only 1 at a time)
highlighted: false  # set true for the 2-3 highlighted entries on homepage
severity: "critical"  # critical | major | minor
categories: ["database", "backup-failure"]  # from canonical list below, drives homepage filter
root_cause_type: "human-error"  # separate from categories, see enum below
one_line_summary: "An engineer ran rm -rf on production. Five backup strategies all failed."
source_url: "https://about.gitlab.com/blog/2017/02/10/postmortem-of-database-outage-of-january-31/"
reading_time: 12
```

**Canonical categories** (drives homepage filter tags):
`database` | `dns` | `deploy` | `cascading-failure` | `configuration` | `security` | `networking` | `third-party` | `caching` | `load` | `financial`

**Root cause type enum** (separate dimension, not displayed as filter):
`human-error` | `configuration-change` | `deploy-failure` | `cascading-failure` | `security-breach` | `third-party-dependency` | `capacity` | `latent-bug`

Categories describe WHAT was affected. Root cause type describes WHY it happened. No overlap by design.

### Detail Page Structure

Each MDX file follows this content structure (enforced by editorial guideline, not schema validation):

```mdx
---
(frontmatter above)
---

## What Happened
(Narrative of the incident timeline. 1-3 paragraphs.)

## Impact
(Blast radius: users affected, duration, data loss, financial impact. Be specific with numbers.)

## Root Cause
(Technical explanation of why it happened. Not just "human error" but the system conditions that made the error possible.)

## Lessons
(What changed as a result. What the reader should take away.)
```

The four sections are a content guideline, not enforced by schema. Some postmortems may warrant additional sections (e.g., "Timeline" or "Recovery"). The goal is consistency, not rigidity.

**Enforcement mechanism:** Include a `src/content/postmortems/_template.mdx` file that authors copy when adding new entries. Editorial review during the PR process serves as the quality gate.

**Three card tiers:**
1. **Featured** (1 entry): Full-width, large headline (2rem+), pull quote or key insight, 2-3 sentence summary. Frontmatter: `featured: true`. Only one entry at a time.
2. **Highlighted** (2-3 entries): Two-column layout. Headline + summary + company + severity. Frontmatter: `highlighted: true`.
3. **Standard** (remaining): Three-column compact cards. Headline + company + severity badge. No summary text.

**Collision guard:** Homepage template sorts by `date` descending, takes the first entry with `featured: true` (ignores duplicates). Same for `highlighted: true` (takes first 3 by date). This prevents layout bugs if multiple entries are accidentally marked featured.

**Filter placement:** Below the featured entry, above the highlighted/standard grid. Filters affect highlighted and standard tiers only (featured always shows).

### Interaction States

**Filter mechanism:** URL-based. Clicking "Database" navigates to `/?category=database`. Cards filter with a CSS fade transition (~150ms opacity). Multi-select: no, single category at a time. "All" clears the filter. Active filter tag gets amber background (`#fffbeb`) + amber border. Astro handles this with a small client-side script (`<script>` island) that intercepts clicks and updates visibility without full page reload, while also updating `history.pushState` for shareable URLs.

| Feature | Default | Hover | Active/Selected | Empty | Error |
|---------|---------|-------|-----------------|-------|-------|
| **Filter tag** | Gray border (#d6cfc2), gray text (#777) | Amber border, amber text | Amber bg (#fffbeb), amber border, amber text | N/A | N/A |
| **Standard card** | White bg, subtle border (#e8e3da) | Border darkens to #c9c1b3, subtle box-shadow (0 2px 12px rgba(0,0,0,0.04)) | N/A (cards are links, not toggles) | N/A | N/A |
| **Featured card** | No border, full-width | Subtle text color shift on headline | N/A | N/A | N/A |
| **Card link** | Entire card is clickable (wrap in `<a>`) | Cursor pointer, card hover state above | Focus: 2px amber outline offset 2px (keyboard nav) | N/A | N/A |
| **Empty filter** | N/A | N/A | N/A | "No postmortems in this category yet. Check back soon, or browse all." + link to clear filter. Centered, warm tone, amber accent on "browse all" link. | N/A |

**Site header:** Logo/site name (left), "About" link (right). Minimal. No hamburger on mobile, just collapse "About" to a small link.

**Site footer:** Tagline, RSS link, GitHub repo link, About link. One line on desktop, stacked on mobile.

### Pages
1. **Homepage** — Editorial hierarchy (featured + highlighted + standard tiers), category filters, site header/footer
2. **Detail page** — Full postmortem narrative (see Detail Page Structure above), source link, "Back to all postmortems" link at top
3. **About page** — One-column layout (same 700px reading column as detail page). Content: (a) What this is: a curated collection of the best public postmortems. (b) Why: developers learn from each other's incidents. (c) How entries are selected: read widely, pick the most compelling and educational. (d) Who: maintained by [your name]. (e) How to suggest: link to GitHub issues or email. Keep it short, 3-5 paragraphs.
4. **404 page** — Centered on the page. Headline: "Postmortem not found." Subtitle: "This incident has no write-up. (Or you mistyped the URL.)" Link: "Browse all postmortems" in amber. Same warm tone as the rest of the site.

**Confirmed hero copy:**
- H1: "Where's the **Postmortem**?" (with "Postmortem" in amber accent)
- Subtitle: "Curated incident stories from the best engineering teams. Learn from failures you didn't have to live through."
5. **RSS feed** — Auto-generated via `@astrojs/rss`. Developers expect RSS on content sites. v1 requirement.
6. **Sitemap** — Auto-generated via `@astrojs/sitemap`. One integration in `astro.config.mjs`. Helps Google discover pages faster.

### Responsive Design

Three breakpoints:

**Mobile** (< 640px):
- **Featured:** Full-width, headline 1.6rem, pull quote stays, summary stays. Padding 1rem.
- **Highlighted:** Single-column stack. Same card content as desktop but full-width.
- **Standard:** Compact list view (no grid). Each entry: company + headline on one line, severity badge. No summary text. Tighter vertical spacing.
- **Filters:** Horizontal scroll with overflow. No wrapping. Show 4-5 tags visible, scroll for more.
- **Detail page:** Reading column fills viewport with 1rem padding. Body text 1rem. Headings 1.8rem.
- **Header:** Logo left, "About" link right. Single line.
- **Footer:** Stack vertically. Tagline on top, links below.

**Tablet** (640px-1024px):
- **Featured:** Full-width, headline 2rem.
- **Highlighted:** Two-column grid.
- **Standard:** Two-column grid.
- **Detail page:** 100% width with 2rem padding.

**Desktop** (> 1024px):
- **Featured:** Full-width, headline 2.4rem+.
- **Highlighted:** Two-column (each ~50%).
- **Standard:** Three-column (auto-fill, minmax 280px).
- **Detail page:** 700px max-width centered.

### Accessibility Specifics

- **Keyboard navigation:** All interactive elements (cards, filter tags, source links) must be focusable via Tab. Focus order: header nav -> featured card -> filter tags -> highlighted cards -> standard cards -> footer.
- **Focus indicators:** 2px solid amber outline (#b45309) with 2px offset. Never `outline: none` without a visible replacement.
- **ARIA landmarks:** `<header role="banner">`, `<main role="main">`, `<nav role="navigation">` for filter bar, `<footer role="contentinfo">`.
- **Card semantics:** Each card is an `<article>` wrapping an `<a>`. Screen reader announces article title.
- **Filter bar:** `role="toolbar"` with `aria-label="Filter by category"`. Active filter gets `aria-pressed="true"`.
- **Touch targets:** All clickable elements minimum 44x44px on mobile. Filter tags need adequate padding.
- **Skip link:** Hidden "Skip to content" link at the top for keyboard users. Skips to `<main>`.

## User Journey

| Step | User Does | User Feels | Design Supports It |
|------|-----------|------------|-------------------|
| 1 | Clicks link from Slack/HN/Twitter | Curious. "What is this?" | Hero headline + subtitle answers immediately. No ambiguity. |
| 2 | Sees featured postmortem | Intrigued. "Oh, THAT incident." | Full-width featured entry with compelling headline and pull quote. Recognition of a famous incident hooks them. |
| 3 | Scans highlighted + standard entries | Browsing. "There's more." | Tiered layout gives visual rhythm. Severity badges add urgency. Categories help orient. |
| 4 | Clicks a postmortem | Anticipation. "Tell me the story." | Card-to-detail transition is instant (static site). Clean reading column. |
| 5 | Reads through the four sections | Absorbed. "Whoa, five backups all failed." | 700px column, 1.8 line-height, Source Serif 4. Nothing competes for attention. |
| 6 | Finishes reading | Reflective. "I should check our backups." | Source link to original. Back link to homepage. Maybe reads another. |
| 7 | Returns to homepage | Hungry for more. "What else happened?" | Filters help them find what they care about. |
| 8 | Shares the link | Generous. "My team needs to see this." | Clean OG image + title make the share look good in Slack/Twitter. |

**The "sold" moment is Step 2.** The featured postmortem must be the single most compelling entry. If a first-time visitor doesn't click through from the featured entry, the homepage failed. Choose a universally recognized incident (GitLab database deletion or Knight Capital) as the initial featured entry.

## Content Licensing Position

Summaries are transformative works under fair use. Each entry links prominently to the original source. We are not reproducing full postmortems, we're writing original narrative summaries (200-500 words) that reference the key facts. This is comparable to a book review or a news summary. The source link drives traffic back to the original, which is a net positive for the publisher.

If any company requests removal of their postmortem summary, comply immediately. Add a note to the entry's frontmatter (`removed: true`, `removal_reason: "company request"`) and exclude it from the build.

## Content Curation Effort

Each postmortem entry requires:
1. Read the original (2,000-10,000 words): ~20 min
2. Write the narrative summary (200-500 words across 4 sections): ~30-60 min
3. Fill frontmatter metadata (categories, severity, root cause, source URL): ~5 min
4. Editorial review (does the summary stand alone? is it accurate? is it engaging?): ~15 min

**Estimated time per entry: 1-2 hours.** For 10-15 seed entries, budget 15-25 hours of editorial work. This is the highest-effort task in the project and should not be underestimated. AI-assisted drafting of summaries is an option, but every summary must be human-reviewed for accuracy and voice.
